    getMftPagination: builder.query({
      query: ({
        page,
        pageSize,
        searchTerm,

        dealDate,
        valueDate,

        sortBy,
        filter,
        showComments,
      }) => {
        const queryParams = queryString.stringify({
          ...(page && { pageCount: page }),
          ...(pageSize && { pageSize }),
          ...(searchTerm && { searchTerm }),
          ...(CounterpartyId && { CounterpartyId }),
          ...(SearchDateFrom && { SearchDateFrom }),
          ...(SearchDateTo && { SearchDateTo }),
          ...(Dealer && { Dealer }),
          ...(currencyId && {
            currencyId,
          }),
          ...(dealDate && { dealDate }),
          ...(valueDate && { valueDate }),
          ...(sortOrder && { Sorting: sortOrder }),
          ...(sortBy && { SortBy: sortBy }),
          ...(givenOrTaken && { givenOrTaken }),
          ...(filter && { Filter: filter }),
          ...(showComments && { showComments }),
        });

        return {
          url: `MutualFundTransfer/GetViaPagination?${queryParams}`,
          method: HTTP_METHODS.GET,
        };
      },
      keepUnusedDataFor: 0,

      query: ({ id }) => {
        return {
          url: `MutualFundTransfer/${id}`,

        url: "MutualFundTransfer/UpdateMutualFundOperation",
        method: HTTP_METHODS.POST,
        data,
      }),

    üÜö Throttle vs. Rate Limit: Major Differences

    The terms API throttling and rate limiting are closely related, but they have distinct meanings and serve different purposes in managing API traffic.

    Feature Rate Limiting Throttling
    Definition Sets a fixed limit on how many requests a client can make in a given time period. Controls the speed or frequency of requests by delaying or rejecting excess ones.
    Purpose Prevent abuse and overuse of APIs by defining usage caps. Smooth out traffic spikes and protect backend resources.
    Behavior Blocks or rejects requests once the limit is hit. May delay or queue requests instead of rejecting them immediately.
    Implementation "You can make 100 requests per hour." "You can make 10 requests per second; extra requests will be delayed."
    Analogy Like a speed limit sign: don‚Äôt exceed this number. Like a traffic light: controls how many cars go through at a time.

    BANK

    In the banking system‚Äîespecially in interbank communication and financial messaging ‚Äî standardized message types are used to ensure secure, structured, and automated communication between banks, financial institutions, clearing houses, and central banks.

    Here‚Äôs a breakdown of the main types of messages used in banking, especially in the context of SWIFT, ISO 20022, and RTGS systems

    üßæ 1. SWIFT Message Types (MT messages)

    SWIFT (Society for Worldwide Interbank Financial Telecommunication) uses a standardized messaging system, identified by MTxxx codes (Message Type).

    Message Type Purpose Example Code
    MT1xx Customer payments and cheques MT103 (Customer Transfer)
    MT2xx Financial institution transfers MT202 (Bank-to-bank transfer)
    MT3xx Treasury markets (forex, swaps, etc.) MT300 (Forex deal)
    MT4xx Collection and cash letters MT400 (Advice of Payment)
    MT5xx Securities markets MT540 (Trade Settlement)
    MT6xx Precious metals and commodities MT600 (Commodity Trade Confirmation)
    MT7xx Documentary credits (Trade Finance) MT700 (Letter of Credit)
    MT9xx Cash management and customer status MT940 (Account Statement)

    üåê 2. ISO 20022 Message Types (MX messages)

    A newer, XML-based messaging standard replacing MT in many systems (e.g. SEPA, SWIFT gpi, TARGET2).

    Category Description Example Message
    pacs. Payments Clearing and Settlement pacs.008 (Credit Transfer)
    pain. Payments Initiation (from customers) pain.001 (Payment request)
    camt. Cash Management camt.053 (Bank Statement)
    acmt. Account Management acmt.023 (Account opening)
    semt. Securities Maintenance semt.002
    reda. Reference Data reda.001

    üîÅ 3. RTGS-Specific Message Types (e.g., Nepal, India, etc.)

    Real-Time Gross Settlement systems often use customized message formats or adapt from SWIFT/ISO 20022:

    Type Usage Example (India‚Äôs RTGS)
    Customer Payment Transfer from one customer to another NEFT/RTGS using MT103 / pacs.008
    Interbank Transfer Between two banks MT202 / pacs.009
    Acknowledgment / Response Confirm transaction Response messages like ACK, NACK, or status updates

    üß† Summary

    System Standard Message Types Example
    SWIFT MT MT103, MT202, MT940
    ISO 20022 MX (XML) pacs.008, pain.001, camt.053
    RTGS Varies Often uses MT/ISO formats
    National Systems Custom or ISO-based e.g. ConnectIPS in Nepal, SEPA in Europe

    If you want examples from a specific country's system (like Nepal‚Äôs RTGS or India‚Äôs NEFT/RTGS), I can provide localized message flow and samples too.

    NEPAL

    In Nepal, banking systems use a combination of ISO 20022 message formats and custom integration messages within platforms like connectRTGS and connectIPS/NPI. Here's a detailed breakdown:

    • connectRTGS uses ISO 20022 for bank-to-bank, high-value RTGS messages.
    • connectIPS/NPI employs JSON-wrapped ISO messages for retail, batch, and scheduled transfers.
    • Status tracking is detailed and structured (statuses like ACSP, RJCT, etc.).

    what i need to know before building banking web application for treasury department

    Building a banking web application for a Treasury Department is a high-stakes, security-critical project. Treasury operations involve cash management, fund transfers, liquidity planning, investments, and risk control, often at institutional or government level. Here's what you must know before you start:

    ‚úÖ 1. Understand Treasury Functions

    Start by understanding what your Treasury Department needs. Common functions include:

    Treasury Function Description
    Cash Management Monitoring balances, liquidity, fund positioning
    Payments & Transfers Initiating interbank transfers (e.g., via RTGS, NEFT)
    Investment Tracking Managing term deposits, government bonds, etc.
    FX and Hedging Handling foreign exchange and derivatives (if applicable)
    Risk Management Tracking exposure to market, credit, and liquidity risk
    Reporting Real-time dashboards, reconciliation, audit trails

    üîê 2. Security & Compliance Requirements

    Treasury applications handle large volumes of money, so:

    üîí Implement strong security controls:

    • Multi-factor authentication (MFA)
    • End-to-end encryption (TLS 1.2+)
    • Role-based access control (RBAC)
    • Audit logging of all transactions

    üìú Ensure regulatory compliance:

    • AML/CFT (Anti-Money Laundering / Counter Financing of Terrorism)
    • KYC (Know Your Customer)
    • Local central bank policies (e.g., NRB in Nepal)
    • Audit & reporting standards (SOX, IFRS)

    üè¶ 3. Banking & Payment System Integrations

    You‚Äôll need to integrate with systems like:

    System Purpose
    RTGS High-value real-time transfers
    NCHL / ConnectRTGS Nepal‚Äôs payment infrastructure
    SWIFT / ISO 20022 For international/multi-bank settlements
    Core Banking System To access account balances, ledger, etc.
    Treasury Management System (TMS) Optional ‚Äî if automating investments/liquidity

    ‚öôÔ∏è 4. Architecture & Technical Decisions

    üíª Core Tech Stack:

    • Frontend: React.js, Angular, or Vue (with strong UI/UX for finance)
    • Backend: Node.js / .NET / Java Spring Boot
    • Database: PostgreSQL / Oracle / MS SQL (with encryption at rest)
    • APIs: REST + secure middleware (SSL, HMAC, JWT)
    • Message Standards: ISO 20022 (pacs.008, camt.053), MT formats (if SWIFT)

    üß± Architectural Musts:

    • Microservices (for scalability and fault isolation)
    • Queue systems (e.g., Kafka or RabbitMQ) for async transaction handling
    • Real-time dashboards (e.g., with WebSockets or SignalR)
    • High Availability and Disaster Recovery (HA/DR)
    • Load balancing and failover mechanisms

    üìä 5. Key Modules You May Need

    Module Name Key Functions
    User & Role Management Access control, audit logs
    Account Management Linking bank accounts, balances
