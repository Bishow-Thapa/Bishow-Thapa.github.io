    Mutex

    Mutex

    A mutex (short for mutual exclusion) is a synchronization mechanism used to ensure that only one process or function can execute a piece of code at a time. In the context of react-context-mutex, it helps prevent multiple React components from running the same asynchronous task simultaneously.

    üîí Analogy

    Think of it like a bathroom key in an office:

    • Without a mutex = everyone runs to the door at once.
    • With a mutex = only one person uses it; others wait their turn.

    üß† Why It's Important in axiosBaseQuery

    • axiosBaseQuery is used as a shared interceptor/fetcher across your app.
    • Multiple components or API slices may hit it at once.
    • If all of them respond to a 401 by refreshing the token at the same time, your app becomes unstable.

    üîÅ Bonus

    You can go even further by letting waiting requests automatically retry after the token is refreshed (since the mutex blocks until the refresh is done).

    Preventing multiple unauthorized errors

    • Using async-mutex to prevent multiple calls to '/refreshToken' when multiple calls fail with 401 Unauthorized errors

    For using a mutex in a React application, you have two popular package options depending on your use case:

    Package Name Description Usage Context
    react-context-mutex A React-specific mutex implementation using React Context to prevent a function from running multiple times concurrently. Ideal for React hooks and components. Works well in browser and NodeJS environments. Best if you want a React hook-friendly mutex that integrates with React context and lifecycle.
    async-mutex A general-purpose async mutex library for JavaScript/TypeScript that provides promises-based locking. Not React-specific but very robust and widely used. Good for any JS async code, including React, but requires manual integration without React context.

    Recommendation for React

    If you want mutex behavior integrated with React hooks and context, use react-context-mutex. It lets you create mutex runners keyed by unique IDs and use them inside hooks to prevent concurrent executions, e.g.:

    import { useMutex } from "react-context-mutex";

const useFetchHook = () => {
  const MutexRunner = useMutex();
  const mutex = new MutexRunner("uniqueKey");

  mutex.run(async () => {
    mutex.lock();
    try {
      const response = await fetch("http://myurl");
      const data = await response.json();
      // process data
    } finally {
      mutex.unlock();
    }
  });
};

    If you want a more general async mutex without React-specific features, or need fine-grained control, use async-mutex:

    import { Mutex } from "async-mutex";

const mutex = new Mutex();

await mutex.runExclusive(async () => {
  // critical section
});

    Installation

    npm install react-context-mutex
# or
npm install async-mutex

    Summary

    Package Name React Integration Complexity Use Case
    react-context-mutex High Simple React hooks/components mutex
    async-mutex Low (general JS) Moderate Any async JS, manual integration

    For React apps, react-context-mutex is often the easiest and most natural choice.

    Question

    Can mutex prevent multiple API hits when receiving a 401 error from one request

    Yes, a mutex can prevent multiple API hits to the refresh token endpoint when multiple requests receive a 401 error simultaneously.

    Explanation based on search results:

    • When several API calls fail with 401 Unauthorized because the access token expired, without a mutex, each failing request might trigger its own refresh token request, causing redundant refresh calls and inefficient network usage.

    • A mutex ensures that only one refresh token request is executed at a time. The first request that detects the 401 acquires the mutex and triggers the refresh token API call.

    • Other requests that also receive 401 while the mutex is locked wait for the mutex to be released (i.e., wait for the refresh call to complete).

    • Once the refresh completes successfully and the token is updated, the waiting requests retry their original API calls with the new token, avoiding multiple refresh calls.

    • If the refresh token call fails (e.g., returns 401 or 400), the mutex logic can be designed to prevent retrying the original requests unnecessarily.

    Examples - queryFn Third-Party SDK

    Using a Third-Party SDK

    Many services like Firebase and Supabase provide their own SDK to make requests. You can use those SDK methods in a queryFn:

    Basic Third-Party SDK
    import { createApi, fakeBaseQuery } from "@reduxjs/toolkit/query/react";
import { supabase } from "./supabaseApi";

export const supabaseApi = createApi({
  reducerPath: "supabaseApi",
  baseQuery: fakeBaseQuery(),
  endpoints: (build) => ({
    getBlogs: build.query({
      queryFn: async () => {
        // Supabase conveniently already has `data` and `error` fields
        const { data, error } = await supabase.from("blogs").select();
        if (error) {
          return { error };
        }
        return { data };
      },
    }),
  }),
});

    You could also try creating a custom base query that uses the SDK, and define endpoints that pass method names or args into that base query.

    Practical

    Sure! I'll update your axiosBaseQuery to use an async mutex with a queue so that:

    • When a 401 Unauthorized response happens,
    • Only the first request triggers the token refresh,
    • All other requests (whether 401 or not) wait until the token refresh completes,
    • Then all queued requests retry with the fresh token.
    import queryString from "query-string";
import { store } from "@shared/store/store";
import { jwtDecode } from "jwt-decode";
import { isEmpty } from "lodash";
import { Mutex } from "async-mutex";
// features
import { updateRefreshToken } from "@features/auth/services/authSlice";
import { showIdleModal } from "@features/idle/services/idleSlice";
// shared
import axiosInstance from "@shared/services/axios/axiosInstance";
import { OAUTH_REFRESH, CONFIG } from "@shared/utils/config";
import { performLogout } from "@shared/utils/logoutHelper";
import { API_STATUS_CODES } from "@shared/utils/apiStatusCodes";
import logger from "@shared/utils/logger";

// Mutex to ensure only one refresh at a time
const refreshMutex = new Mutex();
// Flag to track ongoing refresh
let isRefreshing = false;
// Queue for requests waiting during refresh
let pendingRequests = [];

function enqueueRequest() {
  return new Promise((resolve, reject) => {
    pendingRequests.push({ resolve, reject });
  });
}

function resolvePendingRequests(error = null) {
  pendingRequests.forEach(({ resolve, reject }) => {
    if (error) {
      reject(error);
    } else {
      resolve();
    }
  });
  pendingRequests = [];
}

export const axiosBaseQuery =
  ({ baseUrl } = { baseUrl: "" }) =>
  async ({ url, method, meta, data, params, headers }, api) => {
    // Helper to do the actual request
    const makeRequest = async () => {
      const result = await axiosInstance.base({
        url: baseUrl + url,
        method,
        meta,
        data,
        params,
        headers,
      });
      if (!result || !result.data) {
        throw new Error("Invalid response structure from the server");
      }
      return result.data;
    };

    try {
      // If a token refresh is happening, wait for it before making any requests
      if (isRefreshing) {
        await enqueueRequest();
      }

      const data = await makeRequest();
      return { data, meta: { ...meta, method } };
    } catch (axiosError) {
      const apiStatus = axiosError?.response?.status;
      const detail =
        axiosError?.response?.data?.detail ?? "An unknown error occurred";
      const status = axiosError?.response?.data?.status ?? "Error";
      const title =
        axiosError?.response?.data?.title ?? "An unknown error occurred";
      const type = axiosError?.response?.data?.type ?? "Error";

      if (apiStatus === API_STATUS_CODES.UNAUTHORIZED) {
        // If we are not already refreshing, acquire the mutex and refresh token
        if (!isRefreshing) {
          isRefreshing = true;
          const release = await refreshMutex.acquire();

          try {
            const refreshToken = store.getState().auth.refresh;
            const formData = queryString.stringify({
              grant_type: OAUTH_REFRESH?.GRANT_TYPE,
              client_id: OAUTH_REFRESH?.CLIENT_ID,
              client_secret: OAUTH_REFRESH?.CLIENT_SECRET,
              scope: OAUTH_REFRESH?.SCOPE,
              refresh_token: refreshToken,
            });

            const refreshResult = await axiosInstance.login.post(
              CONFIG.BASE_URL,
              formData,
              {
                headers: {
                  "Content-Type": "application/x-www-form-urlencoded",
                },
              }
            );

            const { data } = refreshResult || {};

            if (!isEmpty(data)) {
              const decodedToken = jwtDecode(data?.access_token);
              api.dispatch(
                updateRefreshToken({
                  access_token: data?.access_token,
                  refresh_token: data?.refresh_token,
                  iat: decodedToken?.iat,
                  exp: decodedToken?.exp,
                })
              );
              // Refresh succeeded, resolve queued requests
              resolvePendingRequests();
            } else {
              logger.error({ msg: "Unable to refresh token. Logged out." });
              performLogout(api.dispatch);
              resolvePendingRequests(new Error("Refresh failed"));
              // Optional: api.dispatch(showIdleModal());
              return { error: { status: 401, data: "Refresh failed" } };
            }
          } catch (error) {
            logger.error({ msg: "Error during token refresh:", error });
            performLogout(api.dispatch);
            resolvePendingRequests(error);
            // Optional: api.dispatch(showIdleModal());
            return { error: { status: 401, data: "Refresh error" } };
          } finally {
            isRefreshing = false;
            release();
          }
        } else {
          // Another refresh is already in progress, wait for it to finish
          await enqueueRequest();
        }

        // After refresh completes, retry the original request
        try {
          const retryData = await makeRequest();
          return { data: retryData, meta: { ...meta, method } };
        } catch (retryError) {
          // Retry failed ‚Äî return error
          return {
            error: {
              status: retryError?.response?.status,
              data: retryError?.response?.data,
            },
          };
        }
      }

      if (apiStatus === API_STATUS_CODES.BAD_REQUEST) {
        return {
          error: {
            status: status,
            data: {
              type: type,
              title: detail,
              detail: title,
            },
          },
        };
      }

      // Return any other errors as is
      return {
        error: {
          status: status,
          data: {
            type: type,
            title: detail,
            detail: title,
          },
        },
      };
    }
  };
    What‚Äôs new here?
    • Added refreshMutex from async-mutex to ensure only one refresh runs at a time.
    • Added isRefreshing flag and pendingRequests queue to block all requests during token refresh.
    • All requests during refresh wait on a promise (enqueueRequest) that resolves only after refresh completes.
    • After refresh, all waiting requests resume and retry with the fresh token.
    • On refresh failure, all waiting requests fail gracefully and logout happens.

    This ensures no API request happens with an expired token during refresh, and you avoid race conditions or multiple refresh attempts.
