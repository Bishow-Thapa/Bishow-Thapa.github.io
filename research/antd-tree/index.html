    Not use Mutating forEach

    Using forEach in the useEffect hook, as shown in your code, is not necessarily bad, but there are other approaches that can be more concise and efficient, especially for transforming and collecting values.

    Reasons to Avoid forEach:
    • Mutating State: forEach is used for side-effects (mutations) and is less declarative compared to array methods like map or filter, which can return a new array without modifying the original.
    • Performance: While forEach is fine for small arrays, if your data set grows, the performance might degrade since you are iterating through each permission in a nested loop. Other methods might be faster or clearer in intention.

    More Declarative Approach:

    Instead of using forEach, you can use reduce or flatMap to create a new array of checkedKeys directly, which is more concise and functional.

    Example with flatMap:

    You can combine the permissions from all groups into a single array and then filter those that are selected.

    useEffect(() => {
  // Use flatMap to flatten the array of permissions and map to their values if selected
  const checkedKeys = responseData.flatMap((group) =>
    group.permissions
      .filter((permission) => permission.selected)
      .map((permission) => permission.value)
  );

  // Set the permissions in the form
  form.setFieldsValue({ permissions: checkedKeys });
}, [responseData, form]);
    Why flatMap is Better Here:
    • Concise and Declarative: flatMap is more declarative than forEach, as it directly transforms the array of permissions into an array of checkedKeys.
    • No Mutation: It doesn't mutate the original array or rely on side effects. Instead, it constructs a new array by mapping and filtering.
    • Flattening: Since flatMap can flatten the results, you don’t need to manually push values into an array.

    Example with reduce:

    You could also use reduce if you prefer a more explicit approach.

    useEffect(() => {
  // Use reduce to accumulate selected permissions' values into checkedKeys
  const checkedKeys = responseData.reduce((acc, group) => {
    group.permissions.forEach((permission) => {
      if (permission.selected) {
        acc.push(permission.value);
      }
    });
    return acc;
  }, []);

  // Set the permissions in the form
  form.setFieldsValue({ permissions: checkedKeys });
}, [responseData, form]);
    Why reduce?
    • Explicit Accumulation: reduce gives you more control over how the array is accumulated, which could be useful if you need more complex transformations in the future.
    • Functional Approach: Like flatMap, reduce is a more functional, immutable way to work with arrays.

    Summary:

    • Performance: Both flatMap and reduce are typically faster than forEach because they allow you to transform data in one pass.
    • Conciseness: flatMap or reduce is more concise and easier to read than manually pushing values inside a forEach.
    • Immutability: These methods create new arrays, which is generally a better practice than mutating existing arrays.

    In your case, using flatMap is the most efficient and readable choice. It avoids unnecessary nested forEach loops, is concise, and directly returns the selected permission value keys in one step.

    Updated & Fixed bug for collapsing initially

    The reason the tree might be collapsed initially, even when you're trying to expand all groups by default, is that the defaultExpandedKeys prop is being set too late or not being applied properly when the component first renders. This could be caused by the following:

    Potential Causes

    1. Asynchronous Data Loading:

      • If responseData is fetched asynchronously (e.g., from an API), it might not be available immediately when the component is first rendered. Since defaultExpandedKeys is set based on responseData, if it's not available at the initial render, the Tree will collapse by default.

    2. Missing or Delayed treeData Update:

      • If treeData is derived from responseData and responseData is changing over time, the Tree component might not react to changes in treeData as you expect. This could cause it to collapse because it hasn’t received the full data yet.
    Solutions

    1. Ensure defaultExpandedKeys is Correctly Set After Data is Loaded:

      • You need to make sure that defaultExpandedKeys is set after the responseData is fully loaded or updated.

      You can handle this by ensuring the defaultExpandedKeys are set dynamically when responseData is available. Here's an example of how you could modify the code to handle this properly.

    2. Use useEffect to Update defaultExpandedKeys:

      • You should use useEffect to set the defaultExpandedKeys once the data is loaded and available.
    Here’s an improved approach
    import { useEffect, useState } from "react";
import { Drawer, Tree, Form } from "antd";
import PermissionFooter from "./PermissionFooter";

const PermissionDrawer = ({
  roleId,
  form,
  handleEditPermission,
  open,
  close,
  loading,
  fields,
  responseData,
}) => {
  const [expandedKeys, setExpandedKeys] = useState([]);

  useEffect(() => {
    // Only set expanded keys when responseData is available
    if (responseData && responseData.length > 0) {
      // Set default expanded keys to include all group keys
      setExpandedKeys(responseData.map((group) => group.groupName));
    }
  }, [responseData]);

  // Format the response data to be used in AntD Tree
  const treeData = responseData.map((group) => ({
    title: group.groupName,
    key: group.groupName, // key for group
    children: group.permissions.map((permission) => ({
      title: permission.value, // Title is the permission value

      checkable: true,
      defaultChecked: permission.selected,
    })),
  }));

  // Function to handle changes in Tree selection
  const onCheck = (checkedKeys) => {
    const permissionValues = checkedKeys.filter(
      (key) => !responseData.some((group) => group.groupName === key) // Ensure group names are excluded
    );
    form.setFieldsValue({
      permissions: permissionValues, // Only set selected permission values
    });
  };

  useEffect(() => {
    // Set initial checked permissions when opening the drawer
    const checkedKeys = responseData.flatMap((group) =>
      group.permissions
        .filter((permission) => permission.selected)
        .map((permission) => permission.value)
    );

  return (

      open={open}
      onClose={() => close()}
      closable={false}
      footer={
        <PermissionFooter
          loading={loading}
          onCancel={close}
          onSubmit={() => form.submit()}

            },

            checkable
            onCheck={onCheck}
            // Ensure all groups are expanded by default
            expandedKeys={expandedKeys} // Control the expanded state
            treeData={treeData}
          />
        </Form.Item>
        {fields}
      </Form>
    </Drawer>
  );
};

    Note

    1. useState for expandedKeys:

      • Introduced a state variable expandedKeys to track which nodes should be expanded. Initially, it's an empty array, but it will be updated once the responseData is available.

    2. useEffect for expandedKeys:

      • The expandedKeys are set inside a useEffect that runs when responseData changes. This ensures that the keys are set once the data is available.

      • The keys for expanding are derived from the responseData, specifically the groupName.

    3. expandedKeys in the Tree Component:

      • The Tree component now uses the expandedKeys state instead of relying on defaultExpandedKeys, which allows dynamic control of the expanded nodes.

    • State-driven Expansion: By controlling the expanded keys via state (expandedKeys), you're ensuring that the tree is only expanded once responseData is fully loaded. This avoids the issue where the tree might collapse if the responseData hasn't been loaded when the component first renders.

    • Efficient Update: The expandedKeys state is updated reactively via useEffect, which makes sure the tree expands only after the data is fully available.

    This solution should solve the issue of the tree being collapsed initially, ensuring all groups are expanded by default as soon as the data is available.

    Collapse when click on dropdown icon

    The reason you can collapse the tree when clicking the icon is likely due to the default behavior of the Tree component in Ant Design. The Tree component, by default, allows users to toggle the expansion of nodes by clicking the expand/collapse icon, even if you set expandedKeys to control which nodes are expanded.

    If you want to prevent users from collapsing the nodes (i.e., make the tree fully expanded and not allow collapse), you can use the following approaches:

    Option 1: Disable the Collapse Behavior Using onExpand

    You can control the expansion of the Tree using the onExpand prop. By providing a custom function for onExpand, you can prevent users from collapsing any nodes.

    import { useEffect, useState } from "react";
import { Drawer, Tree, Form } from "antd";
import PermissionFooter from "./PermissionFooter";

const PermissionDrawer = ({
  roleId,
  form,
  handleEditPermission,
  open,
  close,
  loading,
  fields,
  responseData,
}) => {
  const [expandedKeys, setExpandedKeys] = useState([]);

  useEffect(() => {
    if (responseData && responseData.length > 0) {
      // Set expanded keys to include all group keys (expand all groups by default)
      setExpandedKeys(responseData.map((group) => group.groupName));
    }
  }, [responseData]);

  const treeData = responseData.map((group) => ({
    title: group.groupName,
    key: group.groupName, // key for group
    children: group.permissions.map((permission) => ({
      title: permission.value, // Title is the permission value
      key: permission.value, // key is the permission value
      checkable: true,
      defaultChecked: permission.selected,
    })),
  }));

  const onExpand = (expandedKeys) => {
    // Prevent collapse by resetting the expanded keys to the current state (i.e., do nothing when expanding)
    setExpandedKeys(expandedKeys); // This will still allow expanding, but won't allow collapsing
  };

  const onCheck = (checkedKeys) => {
    const permissionValues = checkedKeys.filter(
      (key) => !responseData.some((group) => group.groupName === key) // Ensure group names are excluded
    );
    form.setFieldsValue({
      permissions: permissionValues, // Only set selected permission values
    });
  };

  useEffect(() => {
    const checkedKeys = responseData.flatMap((group) =>
      group.permissions
        .filter((permission) => permission.selected)
        .map((permission) => permission.value)
    );
    form.setFieldsValue({ permissions: checkedKeys });
  }, [responseData, form]);

  return (
    <Drawer
      title="Edit Permission"
      open={open}
      onClose={() => close()}
      closable={false}
      footer={
        <PermissionFooter
          loading={loading}
          onCancel={close}
          onSubmit={() => form.submit()}
        />
      }
    >
      <Form form={form} layout="vertical" onFinish={handleEditPermission}>
        <Form.Item
          name="permissions"
          label="Permissions"
          rules={[
            {
              required: true,
              message: "Please select at least one permission!",
            },
          ]}
        >
          <Tree
            checkable
            onCheck={onCheck}
            expandedKeys={expandedKeys}
            onExpand={onExpand} // Listen to expansion
            treeData={treeData}
          />
        </Form.Item>
        {fields}
      </Form>
    </Drawer>
  );
};

export default PermissionDrawer;
    Note
    • onExpand Prop: I added the onExpand prop to handle the expansion and prevent the user from collapsing any group. In the provided onExpand function, I make sure that the expandedKeys state is reset to the current value, effectively preventing the collapse action.
    1. User Expands: When a user clicks to expand a node, the onExpand handler is called, and the expandedKeys state is updated.
    2. User Collapses: If the user tries to collapse a node, the expandedKeys will still hold the previously expanded keys, making it impossible to collapse a group.

    This way, the tree will always stay expanded, and users will not be able to collapse any of the groups.

    Option 2: Use expandedKeys Without onExpand

    If you simply want to keep all nodes expanded without allowing collapsing, you can set expandedKeys dynamically and not provide the onExpand handler. The expandedKeys will be controlled solely by the state.

    Example Code (Simplified):
    <Tree
  checkable
  onCheck={onCheck}
  expandedKeys={expandedKeys} // Make sure all groups stay expanded
  treeData={treeData}
/>

    This is simpler, but without the onExpand event, you'll need to ensure that the expandedKeys is always populated with all group keys that need to be expanded.

    Summary:
    • If you don't want users to collapse nodes, use the onExpand prop and prevent the collapse by resetting expandedKeys.
    • If you want full control, use expandedKeys directly without providing an onExpand handler.
