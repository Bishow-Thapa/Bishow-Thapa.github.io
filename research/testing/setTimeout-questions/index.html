    setTimeout Questions

    Here are 10 progressively challenging example questions focused on understanding and using setTimeout in JavaScript — ranging from beginner to advanced levels.

    Example

    🔰 1. Basic Delay

    Q: Log "Hello" to the console after 2 seconds.

    setTimeout(() => {
  console.log("Hello");
}, 2000);

    🔁 2. Multiple setTimeout Calls

    Q: Log "One" after 1 second, "Two" after 2 seconds, and "Three" after 3 seconds.

    setTimeout(() => console.log("One"), 1000);
setTimeout(() => console.log("Two"), 2000);
setTimeout(() => console.log("Three"), 3000);

    🔄 3. Loop with Delayed Logs

    Q: Log numbers 1 through 5, one per second.

    for (let i = 1; i <= 5; i++) {
  setTimeout(() => console.log(i), i * 1000);
}

    🤯 4. What's Wrong Here? (Var Scoping)

    Q: What will this code print?

    for (var i = 1; i <= 3; i++) {
  setTimeout(() => console.log(i), i * 1000);
}

    A: It prints 4 three times. Use let instead of var to fix closure issues.

    🧹 5. Cancel a Timeout

    Q: Schedule a message to log after 5 seconds, but cancel it after 2 seconds.

    const timerId = setTimeout(() => {
  console.log("You won't see this");
}, 5000);

setTimeout(() => {
  clearTimeout(timerId);
  console.log("Timeout canceled");
}, 2000);

    🕰️ 6. Recursively Repeat setTimeout (Alternative to setInterval)

    Q: Print "Tick" every second, but stop after 5 times.

    let count = 0;
function tick() {
  if (count >= 5) return;
  console.log("Tick");
  count++;
  setTimeout(tick, 1000);
}
tick();

    🧪 7. Delay in an Async Function

    Q: Create a reusable wait(ms) function using setTimeout.

    function wait(ms) {
  return new Promise((resolve) => setTimeout(resolve, ms));
}

async function demo() {
  console.log("Start");
  await wait(2000);
  console.log("2 seconds later");
}
demo();

    🕳️ 8. Throttle with setTimeout

    Q: Prevent a function from being called more than once every 2 seconds.

    function throttle(fn, delay) {
  let ready = true;
  return function (...args) {
    if (!ready) return;
    ready = false;
    fn.apply(this, args);
    setTimeout(() => {
      ready = true;
    }, delay);
  };
}

const log = throttle(() => console.log("Clicked"), 2000);

    🧠 9. Order of Execution (Event Loop Insight)

    Q: What will be printed?

    console.log("A");
setTimeout(() => console.log("B"), 0);
console.log("C");

    A: Output:

    A;
C;
B;

    Because setTimeout(..., 0) is scheduled after the current call stack.

    💡 10. Progressive Timeout Based on Input

    Q: For an array of names, log each name with increasing delay (e.g., 0s, 1s, 2s…)

    const names = ["Alice", "Bob", "Charlie"];
names.forEach((name, index) => {
  setTimeout(() => console.log(name), index * 1000);
});

    Theory

    🔍 setTimeout: How It Works (Behind the Scenes)

    🧠 High-level Process:

    When you call:

    setTimeout(() => console.log("Hi"), 1000);

    Here’s what happens:

    1. JavaScript runs synchronously — it encounters setTimeout, registers the callback with the browser or Node.js environment.

    2. The environment sets a timer for the specified delay (e.g. 1000 ms).

    3. After the timer finishes:

    4. The callback is queued in the task queue (aka macro task queue).

    5. The event loop picks it up only after the current call stack is empty and all microtasks have been processed.

    🧩 Is setTimeout a Microtask or Macro Task?

    ✅ Answer:

    setTimeout is a macro task (aka task or macrotask).

    🧬 Difference Between Microtasks & Macrotasks

    Feature Microtask Macrotask (Task)
    Examples Promise.then, queueMicrotask setTimeout, setInterval, setImmediate, I/O
    Timing Runs after current stack, before any macrotask Runs after microtasks
    Priority Higher priority Lower than microtasks
    Use case Quick, short operations Scheduled tasks or async callbacks

    🔁 Event Loop Execution Order (Simplified)

    Call Stack → Microtask Queue → Macrotask Queue
    1. Finish current synchronous code
    2. Empty the microtask queue
    3. Take one macrotask, run it
    4. Repeat...

    🧪 How to Find If It’s Micro or Macro

    You can experiment:

    console.log("script start");

setTimeout(() => {
  console.log("setTimeout");
}, 0);

Promise.resolve().then(() => {
  console.log("promise1");
});

queueMicrotask(() => {
  console.log("microtask");
});

console.log("script end");

    💡 Output:

    script start
script end
promise1
microtask
setTimeout
    🔎 Why?
    • promise1 and queueMicrotask are microtasks
    • setTimeout is a macro task
    • Microtasks run before macrotasks once the stack is empty

    🧠 Visual Flow

    JS Execution Starts
↓
Script Runs (synchronous)
↓
Queue Microtasks (Promise.then, queueMicrotask)
↓
Run Microtasks (if any)
↓
Run First Macrotask (like setTimeout callback)
↓
Repeat

    🧵 Real-World Implication

    If you're doing something like:

    setTimeout(() => console.log("timeout"), 0);
Promise.resolve().then(() => console.log("promise"));

    Even though both are "async", the promise runs first, because it’s a microtask, which has higher priority than a setTimeout (macro task).

    🛠️ Summary

    • setTimeout is macro task.
    • Use Promise, queueMicrotask for microtasks.
    • Microtasks always run before macrotasks, once the stack clears.
    • To debug or inspect, use logging and DevTools (Performance tab) to track callback execution order.

    🧠 JavaScript Event Loop Explained (Simplified)

    JavaScript has:

    • Call Stack – runs synchronous code.
    • Microtask Queue – runs after the current script but before timers (like setTimeout).
    • Macrotask Queue – for setTimeout, setInterval, etc.

    🧠 JavaScript Event Loop – Deep Dive

    Absolutely! Let's dive into the JavaScript Event Loop with depth and clarity, highlighting where people commonly get confused — and clear those up with visuals and examples.

    JavaScript is single-threaded, meaning it can do one thing at a time. But it's also asynchronous – it feels like it can do multiple things, thanks to the event loop, callback queues, and background APIs.

    🔁 The Event Loop – Core Components

    1. Call Stack (synchronous code runs here)
    2. Web APIs / Background Threads (for async stuff like setTimeout, fetch, etc.)

    3. Task Queues:

      • Microtask Queue (Promise callbacks, queueMicrotask)
      • Macrotask Queue (aka Task Queue – setTimeout, setInterval, UI events)

    4. Event Loop (the engine that runs this whole show)

    ⚙️ How it works – Step-by-Step

    🔹 Step 1: Execute Synchronous Code

    • The call stack runs all regular (synchronous) code top to bottom.
    • Any async tasks (like setTimeout, Promise) are sent to their appropriate queues.

    🔹 Step 2: After the Stack is Empty

    Once the call stack is empty, the event loop:

    1. Checks the Microtask Queue and runs everything in it (before anything else).

    2. Only when the microtask queue is empty, it checks the Macrotask Queue (like setTimeout) and runs one task.

    3. Then goes back to check microtasks again. Repeat.

    🔥 This is where most people get confused — microtasks are always processed before macrotasks, and they are processed in full before any macrotask can run.

    📦 Common Queues: What Goes Where?

    Feature Queue Type Runs when?
    console.log() Call Stack Immediately
    setTimeout Macrotask After all microtasks are done
    setInterval Macrotask Same as above
    Promise.then Microtask After call stack, before macrotasks
    queueMicrotask Microtask Same as above
    fetch().then() Microtask Then/catch is a microtask
    requestAnimationFrame Special case Before paint

    🔍 Example: What trips people up

    🔥 Code:

    console.log("start");

setTimeout(() => {
  console.log("setTimeout");
}, 0);

Promise.resolve().then(() => {
  console.log("promise");
});

queueMicrotask(() => {
  console.log("microtask");
});

console.log("end");

    🧠 What happens:

    Step Action Output
    1 console.log("start") start
    2 setTimeout added to macrotask (queued)
    3 Promise.then(...) → microtask (queued)
    4 queueMicrotask(...) → microtask (queued)
    5 console.log("end") end
    6 Microtasks run: promise
    microtask
    7 Macrotask (setTimeout) runs setTimeout

    ✅ Final Output:

    start
end
promise
microtask
setTimeout

    🧠 Confusing Things Clarified

    ❗ Confusion #1: Why does Promise.then run before setTimeout?

    Because Promise.then goes into the microtask queue, and the event loop always prioritizes microtasks before macrotasks.

    ❗ Confusion #2: Is queueMicrotask the same as Promise.then?

    Almost. Both queue microtasks. The difference is:

    • Promise.then has overhead of creating/handling a Promise object.
    • queueMicrotask is a lower-level, lightweight way to queue a microtask.

    They go in the same queue, and execute in insertion order.

    ❗ Confusion #3: Is setTimeout(..., 0) truly "immediate"?

    No. Even with 0ms delay, it goes into the macrotask queue. It will only run after all synchronous and microtasks are done.

    🛠 Visual Model

    +------------------+
|   Call Stack     | <---- Runs first
+------------------+
        ↓
+------------------+
| Microtask Queue  | <---- Runs after stack is empty
| (Promises, qMT)  |
+------------------+
        ↓
+------------------+
| Macrotask Queue  | <---- setTimeout, setInterval, etc.
+------------------+

    ⚠️ The Event Loop always clears microtasks first before even touching macrotasks.

    🎓 Summary

    • JavaScript runs synchronous code first.
    • Then processes all microtasks (like Promise.then, queueMicrotask).
    • Then processes one macrotask (like setTimeout), then checks microtasks again.
    • This cycle continues forever.
