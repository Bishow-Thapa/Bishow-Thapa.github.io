    When building dynamic UIs in React, especially tables or lists, you‚Äôll often need to conditionally add elements to an array. Two common patterns are:

    • Using .push() to build the array step-by-step
    • Using a declarative .filter(Boolean) pattern

    This post compares the two and shows best practices when working in React.

    Problem: Building Dynamic Columns

    For example, say you have a table and want to show columns based on user settings or permissions:

    const columns = [...baseColumns];

if (showComments) {
  columns.push(commentColumn);
}

if (isDisplayBackOfficeColumn) {
  columns.push(backOfficeCol);
}

    Is push() Safe in React?

    Yes ‚Äî as long as you‚Äôre not mutating React state directly.

    ‚úÖ Safe usage:

    const columns = [...baseColumns]; // Creates a new array
columns.push(commentColumn); // Modifies local array only

    ‚ùå Unsafe usage:

    state.columns.push(commentColumn); // Mutates state directly ‚Äî avoid this!

    Instead, use:

    setColumns((prev) => [...prev, commentColumn]);

    Using filter(Boolean) Instead

    Another approach is to use a declarative array with conditional entries:

    const columns = [
  ...baseColumns,
  showComments && commentColumn,
  isDisplayBackOfficeColumn && backOfficeCol,
].filter(Boolean);
    Pros:
    • Concise
    • Clean for simple conditions
    Cons:
    • Harder to debug
    • Can become unreadable with complex logic

    Which One Should You Use?

    Pattern When to Use
    .push() Use when logic is complex or conditions grow over time
    .filter(Boolean) Use for 1‚Äì2 simple flags or inline logic

    Final Thoughts

    In most React projects, clarity beats cleverness. Start with .push() for better readability and expandability ‚Äî then refactor if needed.

    NOTE

    ‚ö†Ô∏è Note About isDisplayBackOfficeColumn

    This line:

    const isDisplayBackOfficeColumn =
  value[SETTINGS_KEYS.IS_SERIAL_NUMBER_NEEDED] || BOOLEAN_STRINGS.TRUE;

    Is risky, because it always falls back to BOOLEAN_STRINGS.TRUE ("true"), which will make the condition always true, even if the setting is missing or explicitly "false".

    If you're storing booleans as strings ("true" / "false"), compare explicitly:

    const isDisplayBackOfficeColumn =
  value[SETTINGS_KEYS.IS_SERIAL_NUMBER_NEEDED] === BOOLEAN_STRINGS.TRUE;

    ‚úÖ This ensures the column is only added when the setting is actually "true".

    TIPS

    Separate Utils

    Yes, absolutely ‚Äî extracting dynamic column-building logic into a reusable utility function is a great idea, especially if you're working with multiple tables or need consistent logic across your React app.

    Why Move to a Utility?
    • Keeps components clean
    • Centralizes logic (easy to test and update)
    • Reusable across multiple components or pages

    üõ†Ô∏è Example: Utility Function

    Let‚Äôs say you want to build columns based on flags like showComments, showBackOfficeCol, and maybe more in the future.

    1. utils/columnsBuilder.js

      export function buildColumns({
  baseColumns,
  showComments,
  showBackOfficeCol,
}) {
  const columns = [...baseColumns];

  if (showComments) {
    columns.push({ title: "Comments" }); // or `commentColumn`
  }

  if (showBackOfficeCol) {
    columns.push({ title: "NEW BACKOFFICE" }); // or `backOfficeCol`
  }

  // Extend for other conditional columns
  // if (hasPermission(...)) {
  //   columns.push({ title: "Confidential" });
  // }

  return columns;
}

      • Dynamic column configs
      • Role/permission checks
      • Feature flags
      • Memoization (if needed for performance)

    If you want a truly flexible version, use an array of "column conditions" and map through them:

    if (condition) columns.push(column);
  });

}

    QUESTIONS

    Yes ‚Äî as long as you‚Äôre not mutating React state directly. so this means if we should not direct mutate the useState hooks for example const [columns, setColumns] = useState([]). if we can't direct mutate this useState using push but we can direct mutate the local array, is this correct.

    Yes, you've got the right idea ‚Äî let‚Äôs clarify it to remove all ambiguity.

    ‚úÖ React useState: Immutability Is Crucial

    const [columns, setColumns] = useState([]);

    With useState, you must never mutate the state variable directly.

    ‚ùå Do not do this:

    columns.push(newColumn); // ‚ùå directly mutating the state
setColumns(columns); // ‚ùå React might not detect this change

    This mutates the original array in memory, and React may not trigger a re-render because it checks for reference changes, not internal content.

    ‚úÖ Do this instead:

    setColumns([...columns, newColumn]); // ‚úÖ creates a new array reference

    or using .push() safely:

    setColumns((prev) => {
  const updated = [...prev]; // ‚úÖ clone the old array
  updated.push(newColumn); // ‚úÖ modify the new array
  return updated; // ‚úÖ React sees a new array reference
});

    ‚úÖ Local Arrays Are Safe to Mutate

    If you create a new array locally (e.g., not from useState), then mutating it with .push() is totally fine:

    const columns = [...baseColumns]; // new local array

if (condition) {
  columns.push(commentColumn); // ‚úÖ safe to push
}

    Here, you‚Äôre not touching React state ‚Äî you're just building a new array from scratch in your render logic. This is safe, clean, and often the recommended way to conditionally build arrays in React.

    üîÅ Summary

    Scenario Can Mutate? Recommended?
    useState variable (columns) ‚ùå No Use immutable updates
    Local array created via [...] ‚úÖ Yes Safe to mutate
    Props from parent ‚ùå No Never mutate props

    ‚úÖ Rule of Thumb

    Only mutate things you own. If React manages it (like state or props), treat it as immutable.

    üí° Extra Tip: If using Redux or Context

    If you're deriving showComments or isDisplayBackOfficeColumn from Redux or Context, those values might update often ‚Äî so memoization could help avoid unnecessary recalculations even more.
